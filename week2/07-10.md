# be-DailyJob

> 오전: 웹 백엔드 직무 밋업
>
> 오후: H2 + JDBC 데이터베이스 연동

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 매직 넘버 및 중복 리터럴 상수 추출 
  - H2 데이터베이스 연동
  - DatabaseConfig 클래스 생성
  - 기존 테스트 코드 DB 환경 대응 수정

- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 커스텀 예외 계층 설계 — catch 방식 vs 중앙 ExceptionHandler 방식 사이에서 결정을 미룸

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: HashMap → JDBC 전환 시 public 메서드 시그니처를 유지하여 내부 구현만 교체
- 그렇게 판단한 이유: Database.addUser(), findUserById() 등의 시그니처를 유지하면 AuthController, ViewController 등 호출부를 수정할 필요가 없음. 변경 범위를 데이터 계층으로 한정하여 리스크를 줄임
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: Controller 계층 변경 0건. 이후 게시글/댓글 테이블 추가 시에도 같은 패턴으로 일관되게 확장 가능

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 테스트에서 SessionManager.createSession() 호출 시 SQL 에러 발생 — sessions 테이블의 FK 제약(user_id → users) 때문에 유저가 DB에 없으면 세션 INSERT가 실패
- 실제 원인: 기존 테스트의 createLoginSession()은 User 객체만 만들어서 세션에 넣었지만, JDBC 전환 후에는 DB에 유저가 존재해야 FK 제약을 만족함
- 해결 방법: createLoginSession()에서 Database.addUser(user)를 세션 생성 전에 호출하도록 수정. @BeforeEach에서 테이블 정리 시에도 FK 순서(sessions → users)를 지켜 DELETE 실행
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 인메모리 → DB 전환 시 FK 제약에 의한 데이터 삽입 순서와 삭제 순서를 먼저 확인

# 주간 프롬프트 회고

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
> 현재 코드베이스에서 매직 넘버, 중복 문자열 리터럴, 미추출 상수를 전수 조사하고, 상수 추출이 필요한 항목과 그대로 둬도 되는 항목을 분류해줘

선택한 이유: 대상을 구체적으로 명시하고, 분류 기준까지 제시하여 판단이 포함된 결과를 이끌어냄

### 2. 원본 프롬프트 구조 분석
- 문제 상황: 코드 전반에 매직 넘버와 반복 문자열이 산재해 있어 유지보수 시 변경 누락 위험
- 원하는 결과를 어떻게 요구했는가: "전수 조사"로 범위를 전체로, "분류해줘"로 출력 형식을 지정
- 출력 방식이 명확한가: "분류해줘"로 단순 나열이 아닌 카테고리별 분류를 요청

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
현재 Java 웹 서버 프로젝트의 전체 소스 코드에서 매직 넘버, 중복 문자열 리터럴, 미추출 상수를 전수 조사해줘. 결과를 다음 기준으로 분류해줘: 1) 상수 추출이 필요한 항목 — 반복 사용되거나 변경 가능성이 있는 값, 2) 추출 불필요한 항목 — HTTP 프로토콜 명세, 파싱 구분자 등 그 자체가 표준인 값. 각 항목에 파일명, 라인 번호, 현재 값, 추출 시 상수명 제안을 포함해줘.
```
