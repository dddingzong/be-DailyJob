### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 기술 계층별 패키지에서 도메인 기반 패키지로 전환
- 그렇게 판단한 이유: User 관련 클래스가 3개 패키지에 분산되어 있어 도메인 파악이 어렵고, 기능 추가 시 여러 패키지를 넘나들어야 했음. 도메인 단위로 모으면 응집도가 높아지고 관련 코드를 한 곳에서 파악 가능
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: User 도메인의 모든 클래스가 한 패키지에 모여 코드 탐색이 편해졌고, 범용 인터페이스(`Controller`)는 인프라 계층에 두어 도메인과 분리됨

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `HttpRequest`는 `BufferedReader`, `HttpResponse`는 `DataOutputStream`을 생성자에서 받아 실제 소켓 없이는 객체 생성이 어려웠다
- 처음 추측한 원인: Mockito 같은 mock 프레임워크가 없으면 Controller 테스트가 불가능하다고 생각
- 원인을 확인하기 위해 시도한 방법: `HttpRequest`와 `HttpResponse`의 생성자가 받는 타입을 확인 - 둘 다 추상 스트림 타입이라 메모리 기반 구현체로 대체 가능
- 해결 방법: `StringReader`로 HTTP 요청 문자열을 주입하고, `ByteArrayOutputStream`으로 응답을 캡처하여 `Location` 헤더 값으로 redirect 대상을 검증
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 의존하는 타입이 구체 클래스인지 추상 타입인지 확인. 추상 타입이면 테스트용 구현체로 대체 가능
