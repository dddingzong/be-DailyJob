### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: RequestHandler에 동적 요청 분기를 추가하는 대신, Router + Controller 구조로 분리
- 그렇게 판단한 이유: 회원가입이 추가되면서 RequestHandler의 책임이 과도해짐. 정적/동적 요청 처리를 분리하면 새로운 기능 추가 시 Controller만 구현하면 되므로 확장성이 높아짐
- 고려했지만 선택하지 않은 대안: RequestHandler 내부에 if/else 분기로 경로별 처리 - 기능이 늘어날수록 코드가 복잡해질 것으로 판단하여 제외
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 새로운 동적 경로 추가 시 RouteConfig에 매핑만 추가하면 되는 구조가 되어 유지보수성 향상

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: Thread Pool + Keep-Alive 구조에서 스레드 풀 크기를 1로 줄였을 때, 두 번째 브라우저 요청이 30초간 대기하는 현상. 응답은 즉시 끝나는데 왜 스레드가 반환되지 않는지 의문
- 처음 추측한 원인: 응답이 빨리 끝나면 스레드도 바로 반환될 것이라고 예상
- 원인을 확인하기 위해 시도한 방법: while(true) 루프와 readLine() 블로킹의 관계를 추적
- 실제 원인: while(true) 루프 안에서 응답 후 break하지 않고 readLine()으로 다시 올라가기 때문에, 클라이언트가 연결을 끊거나 30초 타임아웃이 발생할 때까지 스레드가 반환되지 않음. Keep-Alive는 TCP 연결 유지이고, 스레드 점유는 블로킹 I/O 구현 방식의 결과
- 해결 방법: 현재 구조에서는 setSoTimeout(30000)이 스레드 반환의 안전장치 역할.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 스레드가 어디서 블로킹되고 있는지 확인하고, 연결 생명주기와 스레드 생명주기가 종속적인지 독립적인지 파악

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 스레드 풀과 Keep-Alive 동작 방식에 대해 "왜 이렇게 동작하는지"를 코드 레벨에서 깊이 파고든 점
- 그 접근이 도움이 된 이유: Keep-Alive가 네트워크 레벨 개념이고 스레드 점유는 블로킹 I/O 구현 방식의 결과라는 것을 구분할 수 있게 되었다
- 다음에도 반복하고 싶은 점: 동작 결과만 확인하지 않고, 코드 흐름을 따라가며 근본 원인을 이해하는 습관

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 테스트 코드를 작성하지 못한 점. QueryToMap, HttpRequest 등 단위 테스트가 가능한 클래스가 있었으나 기능 구현에 집중하여 후순위로 밀림
- 그렇게 된 원인: 기능 구현과 구조 리팩토링에 시간을 많이 사용
- 다음에는 다르게 해볼 구체적인 행동: 기능 구현과 테스트 작성을 병행하여 진행하기
